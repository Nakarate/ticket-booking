# กันบอท / กันกวาดตั๋ว — อธิบายสำหรับตอบคำถาม

> เอกสารนี้ไว้ตอบคำถาม “ระบบกันบอทยังไง” — มี 2 ระดับ:
> **ส่วนที่ 1** ภาษาคน (ตอบ BU / ลูกค้า) · **ส่วนที่ 2** เทคนิค (ตอบ SA / dev) · **ส่วนที่ 3** Q&A ที่เจอบ่อย

## สรุปสั้น (TL;DR)

เรากันบอทและการกวาดตั๋ว **3 ชั้น**:

1. **เกมทอยเต๋าก่อนสมัคร** (Proof-of-Work) → บอทสมัครทีละแสนบัญชี “ไม่คุ้ม”
2. **ยามห้ามกดรัว** (Rate limit) → สแปมจากที่เดียวรัว ๆ ไม่ได้
3. **1 คนถือได้จำกัด** (Hold cap) → กวาดที่นั่งด้วยบัญชีเดียวไม่ได้

หลักคิดเดียว: **ทำให้คนซื้อจริงแทบไม่รู้สึก แต่บอทที่จะทำเยอะ ๆ เจ็บหนัก จนไม่คุ้มและทำไม่ทันก่อนบัตรหมด**

## ส่วนที่ 1 — อธิบายแบบภาษาคน (สำหรับ BU / ลูกค้า)

### 1) เกมทอยเต๋าก่อนสมัคร

ก่อนสมัครบัญชีได้ เราสั่งให้เครื่องของคนสมัคร **“ทอยลูกเต๋าจนกว่าจะออก 6-6-6”**

- คนซื้อจริง สมัครครั้งเดียว → ทอยแป๊บเดียว ไม่รู้สึกอะไร
- บอทอยากได้แสนบัญชี → ต้องทอยให้ออก 6-6-6 เป็นแสนรอบ → เสียเวลาเป็นวัน ไม่คุ้ม
- ฝั่งเราแค่มองว่า “ออก 6-6-6 จริงไหม” → เห็นปุ๊บรู้ปั๊บ

**หัวใจ: ทำให้สำเร็จมันยาก แต่เช็คว่าทำจริงไหมมันง่าย**

### 2) ยามห้ามกดรัว

มี “ยาม” คอยนับ — ถ้ายิงคำขอถี่เกินไปจากที่เดียวกัน จะโดนบล็อกชั่วคราว บอทเลยสแปมเร็ว ๆ ไม่ได้ ต้องช้าลง

### 3) 1 คนถือได้จำกัด + จองแล้วต้องจ่าย

- 1 บัญชีจองค้างได้จำนวนจำกัด → อยากกวาดเยอะต้องมีหลายบัญชี (ยิ่งแพงขึ้นอีก)
- จองแล้วไม่จ่ายใน 10 นาที → ปล่อยที่นั่งคืนอัตโนมัติ ไม่ให้จับจองทิ้งไว้เฉย ๆ

### รวมกันแล้ว

บอทจะโกงได้ต้อง **ทั้งช้า ทั้งเปลืองเงิน (ค่าไฟ/ค่าเครื่อง/ค่าพร็อกซี)** — พอช้าขนาดนั้น **บัตรก็ขายหมดไปก่อนที่บอทจะทำเสร็จ** (flash sale หมดใน 2–3 นาที) → เศรษฐศาสตร์ของการโกงพังไปเอง

## ส่วนที่ 2 — เทคนิคจริง (สำหรับ SA / dev)

### 1) Proof-of-Work — เทคนิคคือ SHA-256 hashing

**Hash คืออะไร:** เหมือน “เครื่องปั่นข้อความ” — ใส่ข้อความอะไรลงไปก็ได้ ออกมาเป็นรหัสความยาวคงที่ (เช่น `a3f9c2…`) โดยที่ (ก) เปลี่ยน input นิดเดียว output เปลี่ยนหมด (ข) เดาย้อนกลับไม่ได้ SHA-256 มีอยู่ในทุกที่ (เบราว์เซอร์, มือถือ, เซิร์ฟเวอร์)

**“ทอยเต๋าให้ออก 6-6-6” จริง ๆ คือ:** หาเลข `nonce` ที่ทำให้ `sha256(challenge + ":" + nonce)` มี **บิตศูนย์นำหน้า ≥ ระดับความยาก** เดาทางไม่ได้ว่า nonce ไหนจะได้ → ต้องลองไปเรื่อย ๆ (นี่คือส่วนที่เปลืองแรง)

**Protocol (challenge–response) 5 สเต็ป:**

```
1. Client → POST /api/register {email, password}
2. Server → 400 { error:"pow_required", challenge, difficulty }
           + เก็บ pow:<challenge> ใน Redis (TTL 120 วิ)
3. Client  ลอง nonce = 0,1,2,… จน sha256("<challenge>:<nonce>")
           มีบิตศูนย์นำหน้าครบ → ได้ solution
4. Client → POST /api/register {…} + header X-PoW-Challenge, X-PoW-Solution
5. Server  GETDEL pow:<challenge> (ใช้ครั้งเดียว) + เช็ก hash → ผ่าน = สมัครต่อ
```

**เหตุผลของ design:**

| ทำ | เพราะ |
|---|---|
| Server เป็นคนออก challenge | กัน client เลือกโจทย์ง่าย/เตรียมคำตอบไว้ล่วงหน้า |
| ใช้ครั้งเดียว (`GETDEL` atomic) | กัน replay — แก้ทีเดียวเอาไปยิงซ้ำไม่ได้ |
| TTL 120 วิ | กัน precompute คำตอบกองไว้ก่อนเปิดขาย |
| นับเป็น “บิต” ศูนย์ | ปรับความยากละเอียด: +1 บิต = แพงขึ้น 2 เท่า |

**ตัวเลขจริง (วัดมา):**

| | ค่า |
|---|---|
| ระดับความยาก 18 | บอทเดา ~260,000 ครั้ง ≈ **0.5 วิ/บัญชี** |
| 100,000 บัญชีปลอม | ≈ **14 ชั่วโมง CPU** |
| ฝั่งเราตรวจ | **1 hash** (~ไมโครวินาที) |
| ปรับ +4 บิต | บอทแพงขึ้น **16 เท่า**, เราเท่าเดิม |

> เทคนิคเดียวกับ **การขุด Bitcoin** เป๊ะ — เครื่องขุดคือทอย SHA-256 จนได้รหัสขึ้นต้นด้วยศูนย์ นั่นคือ Proof-of-Work เราแค่เอามาใช้กันบอทแทนขุดเหรียญ
>
> โค้ด: `api/pow.go` · เปิด/ปิดด้วย env `POW_DIFFICULTY` (default 0 = ปิด)

### 2) Rate limit — Redis token bucket

- Token bucket เขียนด้วย Lua ใน **Redis** → atomic และ **ใช้ร่วมกันทุกเซิร์ฟเวอร์** (ไม่ใช่นับแยกต่อเครื่อง)
- จำกัดต่อ IP; เข้มเป็นพิเศษที่ `/api/login` + `/api/register`
- เกินโควตา → `429 Too Many Requests`
- โค้ด: `api/ratelimit.go` · env `RATE_*`, `AUTH_RATE_*`

### 3) Hold cap (กันกวาด/hoarding)

- ก่อนจอง เช็คว่าผู้ใช้คนนี้ถือที่นั่ง PENDING (ยังไม่จ่าย) อยู่กี่ที่ ถ้าเกิน `MAX_HELD_SEATS_PER_USER` → `429`
- โค้ด: `api/handlers.go` (`createBooking`)

### bcrypt ≠ Proof-of-Work (จุดที่คนสับสนบ่อย)

ทั้งคู่เป็น “การกวนเลข (hashing)” เหมือนกัน แต่คนละหน้าที่:

| | ใครเสียแรง | ไว้ทำอะไร |
|---|---|---|
| **bcrypt** | **เซิร์ฟเวอร์เรา** | กันรหัสผ่านหลุด (เก็บแบบ hash, ถอดกลับไม่ได้) |
| **Proof-of-Work** | **เครื่องคนสมัคร/บอท** | กันบอทสแปม (บังคับให้เสีย CPU) |

พูดง่าย ๆ: **bcrypt เราจ่าย** (ปกป้องรหัสผ่านลูกค้า) · **PoW ให้บอทจ่าย** (กันมันกวาดตั๋ว) — เกือบตรงข้ามกันเรื่อง “ใครแบกต้นทุน”

## ส่วนที่ 3 — Q&A ที่เจอบ่อย

**Q: บอทฝ่าได้ไหม?**
ได้ ถ้าทุนหนา (GPU / botnet) — PoW **ยกต้นทุนให้สูงขึ้น ไม่ได้ปิดตาย 100%** จุดขายคือ “ซื้อเวลา + ทำให้ไม่คุ้ม” จนบัตรหมดก่อน โปรดักชันจะเสริม CAPTCHA/bot-detection อีกชั้น

**Q: กระทบลูกค้าจริงไหม?**
คนปกติแทบไม่รู้สึก (แก้โจทย์เสี้ยววินาที) มือถือรุ่นเก่าอาจช้าขึ้นนิดหน่อย → เราตั้งความยาก “พอดี” หรือปรับขึ้นเฉพาะตอนโดนโจมตี

**Q: เปิดตลอดเวลาไหม?**
ค่าเริ่มต้น **ปิด** (`POW_DIFFICULTY=0`) เพื่อไม่ให้กระทบตอนโหลดปกติ/เทสต์ → แนะนำ **ไต่ความยากขึ้นเมื่อเจอสัญญาณโจมตี** (signup พุ่งผิดปกติ, 429 เยอะ)

**Q: ต่างจาก CAPTCHA ยังไง?**
CAPTCHA พิสูจน์ว่า “เป็นคน” · PoW พิสูจน์ว่า “เสีย CPU มาแล้ว” — คนละมุม ฟาร์มบอทที่มีคนจริงนั่งกดผ่าน CAPTCHA ได้แต่ก็ยังต้องจ่าย PoW โปรดักชันควรใช้ **คู่กัน**

**Q: ถ้า Redis ล่ม PoW เป็นยังไง?**
challenge เก็บใน Redis → ถ้า Redis ล่มและ PoW เปิดอยู่ การสมัคร/ล็อกอินจะทำไม่ได้ (fail-closed) เป็น trade-off ที่ต้องรับรู้ (ปกติ Redis ทำ HA)

**Q: ทำไมไม่ใส่ PoW ตอนกดจอง (booking) ด้วย?**
หน้า booking ต้องเร็วมาก (hot path) และมี auth + hold cap + rate limit กันอยู่แล้ว → ใส่ PoW เฉพาะจุดสร้างบัญชี/ล็อกอินซึ่งเป็นประตูหลักของบอท
