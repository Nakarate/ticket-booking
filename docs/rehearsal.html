ระบบจองตั๋ว flash-sale ที่ ขายไม่ซ้ำ

บทซ้อมรวมทุกหัวข้อ — อ่านบนมือถือได้ เอาไว้ทวนก่อนพูด

พิตช์ 1 ประโยค: “ระบบจองที่นั่งที่การันตีไม่ขายซ้ำ แม้คนหลักพันกดพร้อมกัน — ปกป้องรายได้และชื่อเสียงแบรนด์”

1 โจทย์

ตอนเปิดขายบัตร คนแห่พร้อมกัน อะไรจะพังได้

ขายที่นั่งซ้ำ

2 คนได้ที่เดียวกัน → ทะเลาะหน้างาน คืนเงิน โดนด่าลงโซเชียล

ระบบล่มตอนพีค

กดพร้อมกันจนค้าง → ขายไม่ได้เลย เสียยอดทั้งรอบ

บอทกวาดตั๋ว

ตั๋วหมดใน 3 วิ ไปโผล่ตลาดขายต่อ แฟนตัวจริงอด

ทุกข้อคือ เรื่องเงินและความเชื่อมั่น ไม่ใช่แค่เรื่องเทคนิค

2 หัวใจ

ที่นั่งเดียว = ขายได้ครั้งเดียว เสมอ

ถึงมีพันคนกดที่นั่งเดียวกันในเสี้ยววินาที ระบบขายได้ใบเดียวเป๊ะ คนอื่นเห็นทันทีว่าเต็ม

  • กันพลาด 2 ชั้นอิสระ — ชั้นหนึ่งมีปัญหา อีกชั้นยังกันได้
  • เลือกฝั่งปลอดภัย — ถ้าระบบมีปัญหาจะกลายเป็น “จองไม่ได้ชั่วคราว” ไม่ใช่ “ขายซ้ำ”

(ศัพท์เทคนิค: Redis กันแรงกระแทก + Postgres unique index เป็นความจริง)

3 ถ้าระบบล่มหน้างาน

เราออกแบบให้ “ล่มแบบปลอดภัย”

กรณีแย่สุดคือ ลูกค้ารอ ไม่ใช่ที่นั่งซ้ำ — เลยหยุดแล้วเปิดใหม่ได้อย่างสบายใจ

จะเกิดอะไร

คนที่จองไปแล้วที่นั่งไม่หลุด · หยุดขายชั่วคราวข้อมูลไม่เพี้ยน · กลับมาขายต่อได้เสมอ

หน้างานทำ 3 สเต็ป

1) รู้ก่อน (ระบบเตือน) · 2) บอกลูกค้า (“ที่นั่งคุณปลอดภัย”) · 3) แก้หรือเลื่อน

4 กันบอท

ทำให้บอทกวาดตั๋ว ไม่คุ้มและไม่ทัน

เกมทอยเต๋าก่อนสมัคร: ก่อนสมัครได้ เครื่องต้อง “ทอยลูกเต๋าจนออก 6-6-6”

  • คนซื้อจริง สมัครครั้งเดียว → ทอยแป๊บเดียว ไม่รู้สึกอะไร
  • บอทอยากได้แสนบัญชี → ต้องทอยเป็นแสนรอบ = เสียเวลาเป็นวัน ไม่คุ้ม
  • ฝั่งเราแค่ดูว่า “ออก 6-6-6 จริงมั้ย” → เห็นปุ๊บรู้ปั๊บ

อีก 2 ชั้นเสริม: ยามห้ามกดรัว (จาก IP เดียว) + 1 คนถือได้จำกัด (กันกวาดด้วยบัญชีเดียว)

สรุป: บอทต้องทั้งช้าทั้งเปลืองเงิน — บัตรหมดก่อนทำเสร็จ

5 ตัวเลขพิสูจน์

ไม่ได้พูดลอย ๆ — วัดมา

1 / 999จำลอง 1,000 คนแย่งที่นั่งเดียว → ขายได้ 1 ใบ, ที่เหลือชน — oversell 0
26×หน้าเว็บดึงที่นั่งเร็วขึ้น 26 เท่า แม้ event ล้านที่นั่ง
18 + 8ชุดทดสอบอัตโนมัติ (ฝั่ง server + คลิกจริงบนเว็บ) เขียวหมด
13 ท่าลองแฮ็กระบบตัวเอง แล้วปิดทุกช่องที่เจอ

6 Trade-offs (ตอบตรง ๆ)

รู้ว่ามีทางลัด — เลือกทำเองเพราะเป็น demo

  • ทำเอง $0 รันโชว์ได้ทั้งระบบในเครื่องเดียว (offline)
  • ขึ้นจริง ค่อยต่อระบบจ่ายเงินจริง + ล็อกอิน/กันบอทระดับองค์กร + เตรียมกำลังเครื่องช่วงพีค
  • ยังไม่รวม (ตั้งใจ): เก็บเงินจริง · e-ticket/เช็คอิน · คืนเงิน — เป็น roadmap

7 Q&A ที่เจอบ่อย

แตะเพื่อดูคำตอบ

bcrypt กับ “เกมทอยเต๋า” ต่างกันยังไง?

bcrypt = เราเป็นคนทำ ไว้ล็อกรหัสผ่านไม่ให้หลุด · เกมทอยเต๋า (Proof-of-Work) = บอทเป็นคนทำ ไว้กันสแปม — เกือบตรงข้ามกันเรื่องใครแบกต้นทุน

บอทฝ่าได้ไหม?

ถ้าทุนหนา (เครื่องแรง) ก็ฝ่าได้ แต่แพงขึ้นมากและช้าลง จนบัตรหมดก่อน — โปรดักชันเสริม CAPTCHA อีกชั้น มันคือ “ทำให้ไม่คุ้ม” ไม่ใช่ “ปิดตาย”

กระทบลูกค้าจริงไหม?

คนปกติแทบไม่รู้สึก (เสี้ยววินาที) มือถือเก่าอาจช้าขึ้นนิด — เราตั้งความยากพอดี หรือเปิดเฉพาะตอนโดนโจมตี

ทำไมมั่นใจว่าไม่ขายซ้ำ?

มีด่านกัน 2 ชั้นอิสระ + พิสูจน์ด้วยการจำลอง 1,000 คนแย่งที่เดียว (ได้ 1 ใบ) และชุดทดสอบอัตโนมัติทุกครั้งที่แก้โค้ด

Redis/ระบบส่วนหนึ่งล่ม เป็นยังไง?

จองใหม่ไม่ได้ชั่วคราว แต่ที่นั่งที่จองไปแล้วปลอดภัย และไม่มีวันขายซ้ำ — หยุดแล้วเปิดใหม่ได้
