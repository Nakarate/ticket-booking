# Demo Run-of-Show — Flash-Sale Ticket Booking

**ผู้ฟังผสม: BU (business) + สาย technical** → เล่า 2 ระดับเสมอ
กติกากับ BU: **นำด้วยคุณค่าธุรกิจ + ของที่เห็นด้วยตา (UI, ตัวเลข)** ก่อนเข้าโค้ด
จุดขายเดียวที่ตอกย้ำตลอด: **ขายตั๋วไม่มีวันเกินจำนวน แม้คนแห่กดพร้อมกันเป็นพัน — และยังเร็ว ไม่ล่ม**

เวลารวม ~15 นาที · ถ้า BU เยอะ/เวลาน้อย เดินแค่ **Act 1 → 2 → 3 → 6** (ข้าม 4–5 ที่เป็น deep-dive)

## 0. ก่อนเริ่ม (ทำก่อนคนดูเข้ามา ~2 นาที — ห้ามทำหน้า BU)

รีเซ็ตให้ seat map สะอาด + วอร์ม stack:
```bash
docker compose down -v && docker compose up -d --build
docker compose ps                         # postgres/redis = healthy
curl -s http://localhost:8080/readyz      # -> ready
curl -s -o /dev/null -w "%{http_code}\n" localhost:3000   # -> 200
```
เปิดเตรียม: แท็บ UI **localhost:3000** + terminal (ย่อไว้ก่อน) + เปิดไฟล์นี้ไว้อีกจอ

> 💡 **สำหรับ BU ให้โชว์ผ่าน "แอปจริง + ตัวเลขสรุป" เป็นหลัก** พับ terminal เก็บเวลาไม่ได้ใช้ อย่าให้ BU จ้องจอดำๆ นานๆ

## Act 1 — ปัญหาเชิงธุรกิจ (พูด, ~2 นาที) 🎯 *เล่าให้ BU อิน*

🎤 **เฟรมแบบ BU:** "เวลาเปิดขายตั๋วคอนเสิร์ตดังๆ คนกดพร้อมกันเป็นหมื่นในวินาทีเดียว ความเสี่ยงที่แพงที่สุดคือ **ขายที่นั่งเดียวกันให้ 2 คน** — พอถึงหน้างานมีคนนั่งซ้ำ = ต้องคืนเงิน, ลูกค้าด่าลงโซเชียล, แบรนด์เสียหาย. หลายเจ้าแก้ด้วยการปิดขายเป็นช่วงๆ ให้ปลอดภัยไว้ก่อน แต่แลกมาด้วยขายช้า เสียโอกาสรายได้"

🎤 **คุณค่าที่ระบบนี้ให้:** "ระบบนี้การันตี **ขายไม่มีทางเกิน** ต่อให้คนกดพร้อมกันเป็นพัน และ **ยังเร็ว ไม่ต้องปิดขาย** — ปกป้องทั้งรายได้และความเชื่อมั่นลูกค้าพร้อมกัน"

🎤 **หนึ่งประโยคสำหรับสาย tech:** "กันด้วย 2 ชั้น — Redis รับแรงกระแทกหน้าบ้าน, Postgres เป็นด่านตัดสินสุดท้าย ต่อให้ Redis ล่มก็ยังขายซ้ำไม่ได้" (ขยายใน Act 4)

🖥️ ที่ **localhost:3000** — เล่าไปทำไป:

2. เลือก 2–3 ที่นั่ง → 👀 ราคารวมขึ้นทันที, ปุ่มจองเปิด
3. กด **"จองที่นั่ง"** → 👀 **นับถอยหลัง 10 นาที** ที่นั่งเปลี่ยนเป็น "มีคนถืออยู่"
   🎤 (BU): "ตอนนี้ระบบกันที่นั่งนี้ไว้ให้ลูกค้า 10 นาที คนอื่นแย่งไม่ได้ ถ้าไม่จ่ายใน 10 นาที ปล่อยคืนอัตโนมัติ"
4. กด **"ชำระเงิน"** → 👀 ขึ้น **PAID** + โผล่ใน "การจองของฉัน"
   🎤 (BU): "จ่ายเสร็จ ที่นั่งเป็นของลูกค้าถาวร — และถ้าเน็ตกระตุกกดซ้ำ ระบบตัดเงินครั้งเดียวเท่านั้น"

**(ไฮไลต์ให้ BU เห็นภาพ) โชว์การแย่งสด:** เปิด incognito อีกหน้าต่าง เป็นลูกค้าคนที่ 2 เลือกที่นั่งที่คนแรกถืออยู่ → กดจอง → 👀 *"มีคนตัดหน้าไปแล้ว 😅 เลือกใหม่ได้เลย"*
🎤 "ลูกค้าคนที่สองไม่ค้าง ไม่เออเรอร์ — ได้คำตอบทันทีให้ไปเลือกที่ว่าง"

## Act 3 — ไม้เด็ด: 1,000 คนแย่งตั๋วใบเดียว (โชว์จอ, ~2 นาที) 🔥 *ตัวเลขที่ BU จะจำ*

🎤 **เฟรมแบบ BU ก่อนรัน:** "ทีนี้จำลองของจริง — สั่งให้ **1,000 คนกดจองที่นั่งเดียวกัน** พร้อมกันเป๊ะ ถ้าระบบพลาดแม้แต่ครั้งเดียวคือขายเกิน"

🖥️ รัน (terminal โผล่แป๊บเดียว) — **รันจากโฟลเดอร์โปรเจกต์**:
```bash
MSYS_NO_PATHCONV=1 docker run --rm --network ticket-booking_default -e API=http://api:8080 \
  -v "$PWD/k6":/k6 grafana/k6 run /k6/race.js
```
> ⚠️ Windows/Git Bash: **ต้องมี `MSYS_NO_PATHCONV=1` นำหน้า** ไม่งั้น Git Bash จะแปลง `/k6/race.js` เป็น `C:/Program Files/Git/k6/...` แล้ว k6 หาไฟล์ไม่เจอ · Mac/Linux ตัด `MSYS_NO_PATHCONV=1` ออกได้

👀 ชี้ 2 บรรทัด แล้วแปลเป็นภาษา BU:
```
booking_success ... 1        ← "ขายได้ 1 ใบเดียว ถูกต้อง"
booking_conflict .. 999      ← "อีก 999 คนได้คำตอบทันที ไม่มีใครค้าง ไม่ล่ม"
```

🎤 **ปิดจ็อบด้วยหลักฐานจากฐานข้อมูล** (ให้ BU เชื่อว่าไม่ใช่แค่แอปบอกเอง):
```bash
docker exec ticket-booking-postgres-1 psql -U ticket -d ticket -c \
"SELECT COALESCE(max(cnt),0) AS max_per_seat, count(*) FILTER (WHERE cnt>1) AS oversold \
 FROM (SELECT seat_id, count(*) cnt FROM order_items GROUP BY seat_id) x;"
```
👀 `max_per_seat = 1`, `oversold = 0`
🎤 (BU): "**ที่นั่งทุกใบมีเจ้าของได้ไม่เกิน 1 คน — ขายเกิน = ศูนย์** พิสูจน์ถึงระดับฐานข้อมูล นี่คือหัวใจของงานนี้"

## Act 4 — เบื้องหลัง: ทำไมมันไม่มีทางพลาด (deep-dive, ~2–3 นาที) 🔧 *สาย tech — BU ฟังผ่านๆ ได้*

> 🎤 บอก BU: "ช่วงนี้ลงเทคนิคนิดนึงสำหรับทีมวิศวกร ใครไม่อินข้ามได้ครับ" แล้วเปิดโค้ด 3 จุด:
1. **`api/handlers.go` → `holdScript`** — Redis Lua จองแบบ all-or-nothing (single-thread = atomic)
2. **`db/001_init.sql` → `idx_one_seat_one_order`** — `UNIQUE(order_items.seat_id)` ด่านสุดท้าย
3. **`api/handlers.go` → `payOrder`** — optimistic confirm `UPDATE … WHERE status='AVAILABLE'` เช็กจำนวนแถว ไม่ตรง = rollback

🎤 invariants ที่ออกแบบไว้: กัน deadlock (`sort seat_ids`), idempotent pay, server-side sale gate, ปล่อยที่นั่งด้วยการลบ `order_items`

## Act 5 — Performance & scale (deep-dive, ~2 นาที) 🔧

🎤 **เฟรมแบบ BU:** "ที่ว่าเร็วและไม่ล่ม มาจากการจูน DB — เวลาคนเป็นแสนรีเฟรชผังที่นั่ง ระบบไม่เผา CPU ทิ้ง"
🔧 (tech) `docs/benchmark-results.md`: partial covering index → *Index-Only Scan* (~26× เร็วกว่า seq scan, แตะ 100× เมื่อใกล้ขายหมด) + MVCC/autovacuum tuning บนตาราง seats

🎤 **ความซื่อสัตย์เรื่องตัวเลข** (ถ้าถูกถามเรื่อง latency): ~1 วิที่เห็นใน k6 คือ **การเข้ารหัสรหัสผ่าน (bcrypt) ตอนสมัคร/ล็อกอิน 1,000 คนพร้อมกัน** ไม่ใช่ขั้นตอนการจอง — เขียน caveat ไว้ใน doc แล้ว ส่วน oversell=0 reproduce ได้ทุกครั้ง

## Act 6 — ความปลอดภัย & ความพร้อม (พูด + doc, ~2 นาที) 🎯 *BU สนใจ "ความเสี่ยง"*

🎤 **เฟรมแบบ BU:** "ระบบขายตั๋วถือเงินและข้อมูลลูกค้า เราตรวจความปลอดภัยแบบจำลองการโจมตีจริง"
เปิด `docs/security-review.md`:
- **Auth จริง**: รหัสผ่านเข้ารหัส bcrypt + token อายุสั้น + เพิกถอนได้
- **ภัยที่กันอยู่ (ทดสอบแล้ว)**: SQL injection, ปลอม token, แก้ราคาฝั่ง client, ดูออเดอร์คนอื่น (IDOR), ขายเกิน
- **รอบรีวิวล่าสุดแก้เพิ่ม**: ปิดช่อง rate-limit bypass, ผูก idempotency กับออเดอร์, ล็อก DB ไว้ localhost, อัปเดต Next.js, ลบ dead code
- 🎤 (BU): "ทุกช่องที่เจอถูกปิดแล้ว ที่เหลือเป็นงาน production (payment gateway จริง, TLS) มีบันทึกไว้หมด"
